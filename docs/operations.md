# 運用観点

## 概要

このドキュメントでは、Linux / Windows Server と内部ロードバランサーを含む AWS 基盤を運用する際の考え方を整理します。

<br>

## 接続経路

| 経路 | 考え方 |
| --- | --- |
| 管理者接続 | 踏み台サーバーを経由し、Private Subnet 内のサーバーへ接続する |
| アプリケーション通信 | 内部ロードバランサーを接続先とし、個別サーバーへの直接依存を避ける |
| DB接続 | Linux / Windows Server から DB 層への通信のみ許可する |
| 外部通信 | 必要に応じて NAT Gateway 経由で OS 更新や外部サービス連携を行う |

<br>

## メンテナンス

内部ロードバランサー配下の構成では、サーバー単位のメンテナンスを行いやすくなります。

1. 対象サーバーを振り分け対象から外す
2. 対象サーバーで OS 更新、アプリケーション更新、再起動などを行う
3. 動作確認後、振り分け対象に戻す
4. 残りのサーバーも同様に順次対応する

<br>

## 障害時の考え方

| 障害箇所 | 想定対応 |
| --- | --- |
| Linux / Windows Server 単体 | 内部LBのヘルスチェックにより、異常なサーバーを振り分け対象から外す |
| 踏み台サーバー | 管理経路への影響を確認し、代替接続経路または復旧手順を実施する |
| NAT Gateway | 影響AZ、ルートテーブル、外部通信の必要性を確認する |
| DB接続 | アプリケーション層からDB層への通信、認証、Security Groupを確認する |
| 外部通信 | NAT Gateway、ルートテーブル、名前解決、Security Groupを確認する |

<br>

## セキュリティ

- Linux / Windows Server は Private Subnet に配置し、インターネットから直接到達させない
- 管理用通信は踏み台サーバー経由に限定する
- Security Group は役割単位で分離する
- DB層への通信はアプリケーション層からの必要なポートに限定する
- 認証情報や環境固有の接続先はリポジトリに含めない
