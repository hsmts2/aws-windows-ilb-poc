# 運用観点

## 概要

このドキュメントでは、Windows Server と内部ロードバランサーを含むAWS基盤を運用する際の考え方を整理します。

公開用のケーススタディであるため、実環境の接続先、手順、通知先、認証情報は記載しません。

## 接続経路

| 経路 | 考え方 |
| --- | --- |
| 管理者接続 | 踏み台サーバーを経由し、Private Subnet内のサーバーへ接続する |
| アプリケーション通信 | 内部ロードバランサーを接続先とし、個別サーバーへの直接依存を避ける |
| DB接続 | アプリケーションサーバーからDB層への通信のみ許可する |
| 外部通信 | 必要に応じてNAT経由でOS更新や外部サービス連携を行う |

## メンテナンス

内部ロードバランサー配下の構成では、サーバー単位のメンテナンスを行いやすくなります。

1. 対象サーバーを振り分け対象から外す
2. 対象サーバーでOS更新、アプリケーション更新、再起動などを行う
3. 動作確認後、振り分け対象に戻す
4. 残りのサーバーも同様に順次対応する

## 障害時の考え方

| 障害箇所 | 想定対応 |
| --- | --- |
| Windows Server単体 | 内部LBのヘルスチェックにより、異常なサーバーを振り分け対象から外す |
| 踏み台サーバー | 管理経路への影響を確認し、代替接続経路または復旧手順を実施する |
| DB接続 | アプリケーション層からDB層への通信、認証、Security Groupを確認する |
| 外部通信 | NAT、ルートテーブル、名前解決、Security Groupを確認する |

## セキュリティ

- Windows ServerはPrivate Subnetに配置し、インターネットから直接到達させない
- 管理用通信は踏み台サーバー経由に限定する
- Security Groupは役割単位で分離する
- DB層への通信はアプリケーション層からの必要なポートに限定する
- 実運用の認証情報、接続先、通知先はリポジトリに含めない

## 公開リポジトリで扱わないもの

- 実際の運用手順書
- 実環境の障害対応フロー
- 監視通知先
- 接続元IP
- アカウントID
- インスタンスID
- 認証情報
